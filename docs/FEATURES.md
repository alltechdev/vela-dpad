- [x] **Foreign-script street names are spoken and shown romanized (upstream issue #184).** Nav guidance is generated in the app language, but the road NAME comes from the map in its local script, so a Latin-script voice had no phonemes for it and **dropped the name entirely** - exactly the words the driver needs. It matters here more than upstream: the fork ships an `iw` locale and its testers are a largely Hebrew-speaking community. `SpokenScript` romanizes only the runs the speaking voice cannot read (a Hebrew name is romanized for an English, Russian or Chinese voice alike; a Russian voice keeps its own Cyrillic), leaving Latin text, accents, digits and punctuation alone. CJK is deliberately never romanized - ICU reads Han as pinyin, a confidently-wrong reading - so kanji/kana are left to the native CJK voices. **Real names, not a transliteration:** ICU alone produced `'rlwzwrwb` for `ארלוזורוב` on device, so `VelaMapView` harvests each road's `name:en`/`name:latin` off the basemap while navigating (camera-idle, only for roads not already held) and `VoiceGuide` speaks **"Arlosoroff"**. The banner and the "then" line use the same dictionary, with no ICU fallback on screen - a name with no real romanization keeps its local script, because a skeleton beside a Hebrew street sign reads broken. A Hebrew-reading user keeps Hebrew throughout. Also holds the nav-start announcement briefly so the opener says the real name rather than a skeleton at T=0, and never stays mute. **Works offline too:** each routing region publishes a gzipped `<local>\t<English>` road-name sidecar beside its graph (Israel & Palestine is 382 KB / 26,102 entries), downloaded with the graph and merged into the same dictionary at nav start, so a drive with no signal still says the real street name; the tile harvest merges on top when there is a connection. Device-verified in Tel Aviv at 480x854 @320 and 240x320 @160, standard + restricted.
- [x] **Hebrew open/closed status (upstream).** Google PREFIXES its Hebrew status strings with "המקום" ("the place"), so the bare `סגור`/`פתוח` keywords never startsWith-matched and **every Hebrew place showed no open/closed colour at all**. Both word tables carry the prefixed forms, pinned by tests. Ported selectively: this fork's `CalibrationStore` already parsed the word tables upstream was fixing (and two more besides), so ours was kept. Search also stops biasing to null island when a device has never had a GPS fix.
- [x] **Hardware soft keys on keypad phones (2026-07-17).** Feature/kosher/keypad phones carry two hardware SOFT keys with no touch equivalent; issue #65 asked them to drive the map. The vendored [Yapchik](https://github.com/theOnionsAreWatching/yapchik) engine (`:yapchik`, LGPL-3.0, zero deps) renders a bottom LEFT/RIGHT label bar driven by the physical keys. **The two keys are CONTEXTUAL:** bare map = Options / Search (Options opens a feature-phone bottom-left bordered menu - Recenter / Layers / Park / Settings, plus Search this area when the map has been panned, plus one adaptive map-motion item that nests: Move map when idle → Zoom once you're moving → Stop zoom while zooming, with D-pad LEFT/RIGHT panning or zooming and BACK peeling one mode at a time); a place sheet = Options / Directions (Close is BACK); choose-on-map = Cancel / Set start|stop; a previewed route = Steps / Start (Start begins the drive); turn-by-turn = Options / Zoom, where the nav Options menu (Mute/Unmute, Steps, Search along route, Recenter, End) gives a keypad driver full control of the drive; the search overlay shows no bar; Settings suppresses it. **When a soft key covers an action the redundant on-screen button is dropped** (the map's +/-, the whole place-sheet button row + header star/share/overflow) to reclaim space; the Options menu only shows build-applicable items (no Street View / Website in restricted, degoogled geo: Share). A modal `VelaDialog` hides the bar while it's up (its keys go to the dialog); a Settings toggle (`Yapchik.mode`) + a press-your-keys **calibration** row handle on/off and non-standard keycodes. The **bar follows the in-app Light/Dark theme** (rebuilt only on an actual flip so it never flashes on screen changes; `navGuardDp=0` keeps it single-height on framework-themed windows - both tester-confirmed on hardware), and labels are **localized** (Options / Search / Recenter / Mute / Unmute in all 14 locales; Directions / Layers / Settings / Start / Steps reuse existing strings). Everything gates on Vela's conservative D-pad detector so a touch phone is byte-identical. Device-verified on all four committed geometries (480x854 @320 and @225, 240x320 @160 and @120), standard and restricted: 27/27 on every standard leg, 25/25 on both restricted legs (Street View is n/a there). The touch legs are a harness gap, not a clean pass - see docs/softkeys.md. Design + remaining follow-ups (device-keycode XML needs hardware dumps; extra nav sets held by choice) in [`docs/softkeys.md`](softkeys.md).
- [x] **Bare-map declutter under soft keys (tester @SILB, #76).** On a keypad phone the bare map now sheds the on-screen search bar, the Layers button and the Park button the same way the zoom keys already moved: the **RIGHT** soft key opens Search (the collapsed bar is gone; the overlay it opens still carries the mic + gear), and **Layers + Park** join the bare-map **Options** menu (LEFT). With the search bar gone the category chips **ride up** flush under the status bar to reclaim its slot. Park keeps its full hub from the menu - *Save parking spot* with no spot, *Parked car* → Find my car / Move / Earlier / Clear once set (the hub + history sheet are hoisted out of the now-hidden FAB so the menu drives them). The bare-map update card's actions likewise ride the bar (Not now / Update). All gated on `VelaSoftkeys.isActive()`, so touch + hybrid keep the on-screen search bar, Layers button and Park FAB byte-identical. Device-verified 240x320 + 480x854, standard + restricted.
- [x] **Whole-app declutter under soft keys (#76 round 2).** The same rule applied to every remaining surface: a control is dropped only where its action already lives on a soft key, on hardware BACK, or in an Options menu, so nothing is lost and the screen is handed back. **In-nav** loses the ETA card's Mute/Steps/End buttons and both right-edge FABs, leaving banner + map + a clean ETA (which also un-breaks the headline the button row used to squeeze into an ellipsis); the nav Options menu gains **Search along route**, so all five drive actions live there. **Route preview** loses Start + Steps (both are the soft keys) and the top card's back arrow (BACK), handing the width back to the endpoint rows and the route alternatives. **Choose-on-map** loses its Cancel X and Set start/stop button - both were triple-covered by the bar, BACK and OK-on-crosshair, and both were focus stops competing with the panning the surface exists for. The **turn list** loses its close X and gains a working LEFT key (it was bound to `null` - a dead labelled key on the one surface still drawing its own Close). The **bare map** loses the locate FAB and "Search this area" to Options (Recenter re-arms free-drive follow exactly as the FAB tap did), the **resume-nav** card's Dismiss/Resume move onto the bar (they sat in a Row, where LEFT/RIGHT are soft keys and not focus traversal - the same trap as #74), the **search overlay** loses its Settings gear (already in map Options), and the **stops editor** loses its close X (BACK). Two real bugs fell out of the audit: the **arrival** screen had no soft-key branch at all, so the bar offered *Start* - a drive to the place you had just reached - and BACK could not dismiss it (`clearRoute` never clears `arrived`); and **Street View** inherited the place sheet's keys under the panorama, so LEFT set state nothing rendered and RIGHT started routing from behind the pano. Both now bind correctly (Done; Close). Device-verified 480x854 + 240x320, standard + restricted.
- [x] **D-pad focus-ring polish (tester @SILB, x320).** The orange ring now **hugs the control** instead of Material's touch target: `minimumInteractiveComponentSize()` pads a chip's layout to 48dp while it draws at 32dp (buttons 40dp), so a ring on the control's own modifier measured the inflated box and floated clear of it. `DpadRingBox` rings a parent pinned to the real height and `hasFocus` propagates from the child, which keeps its full touch target. The map category chips also get their true `CircleShape` (they were pills ringed with an 8dp rounded rect). Rows no longer draw **two** focus signals - `dpadClickable` drops Material's grey focus state layer while input is key-driven and keeps the press ripple for touch. And a menu toggle row now rings the **switch**, not the whole option, matching Settings.
- [x] **Nav chrome compacted on feature-phone screens.** The in-nav ETA card is now a single thin line under soft keys ("2 min · 0.3 mi · 2:09 AM") instead of a two-line block: with the buttons gone there is width to spare, the duration keeps its intrinsic width and is never truncated, and the tail ellipsizes from the right so a long "1 hr 25 min · 45.3 mi · 11:47 PM" degrades gracefully at 240dp. The **maneuver banner** drops from five lines to three: the distance and the route shields share one line (they were two of the banner's four lines before the instruction even started), paddings and corner radius step down, and the instruction then fits on one line instead of wrapping. Nothing is dropped from either. **This also fixed a silently-dead gate:** the banner's compact mode was keyed on `screenHeightDp < 500`, but `AdaptiveDensity` shrinks a narrow screen's density and inflates the reported dp to match, so a Sonim x320 that is physically 427dp tall reports 640dp and never tripped the test - compact mode had never once run on our primary target device. It now keys off `AdaptiveDensity.applied`. Device-verified on all four committed profiles: 480x854 @320 and @225, 240x320 @160 and @120.
- [x] **Dialog buttons: filled pill confirm, honest voice offer (upstream 837c7b00 + 605ade01).** Every VelaDialog renders its confirm as a filled pill and its decline as a plain text button, and the row wraps (FlowRow) so a long label like "Download Vela Voice" takes its own line instead of breaking mid-word on small screens. The first-run voice offer is a real two-way choice: a prominent Download Vela Voice against a quiet, fully working "Use system voice", with the body split into two short paragraphs. Fork fix on top: the focused filled pill wears a contrasting (onPrimary) focus ring - the default primary ring was invisible on the primary fill. Verified with D-pad walks and the small-screen dialog audit at 240x320, native and 480x854, both flavors.

- [x] **A dropped pin stays where you pressed (user report, 2026-07-16).** Long-pressing the map sometimes moved the camera somewhere else a beat later, touch and D-pad alike: the reverse-geocode that names the pin also carried Google's snapped location (the nearest ADDRESSABLE point, possibly another street), and adopting it re-aimed the already-framed camera. - [x] **Vibrate-on-turns chips wrap instead of clipping (user report, 2026-07-16).** The four travel-mode chips lived in a horizontally scrollable row; on a narrow screen or low density the fourth chip rendered partially cut with nothing hinting the row scrolls. - [x] **Focus follows a Settings control through its swaps (user report, 2026-07-16).** Pressing a Download button used to teleport the highlight to the top of the page: the button is replaced by a progress readout, Compose clears focus with the disposed node, and recovery restarts at the first row. - [x] **The bare-map update/notice cards are D-pad reachable (user report, 2026-07-16).** The Update/Not now buttons (and a pushed notice's Learn more/Dismiss) could never be highlighted with keys: no traversal path reaches the card stack because the search bar above it opens on focus and swallows the way down. The top actionable card now takes focus itself - immediately on a keypad-first phone, and on a hybrid the first BACK out of the search overlay lands on it - via the new reactive `dpadModeAutoFocus()` (the static-device variants no-op on hybrids by design). Buttons wear explicit rings (a TextButton's stock focus overlay is invisible on the tinted card), acting on a card hands focus to the map target, and the category chips the cards fully cover drop out of the focus order (an occluded chip used to take focus invisibly and OK fired its search). Device-verified on a hybrid keypad phone: reach, LEFT/RIGHT between buttons, DOWN to the map and UP back, dismiss handoff.
- [x] **In-app Street View (upstream streetview-inapp, ported 2026-07-16).** The place sheet's Street View pill opens a keyless panorama IN the app - no hand-off to Google's app, no WebView. `GeoPhotoService.SingleImageSearch` resolves the nearest pano (keyless, referer-authorised), tiles come from `streetviewpixels-pa.googleapis.com` and stitch onto a GLES20 equirectangular sphere (`streetview/PanoramaView`). Half-screen over the map with a rotating pegman view-cone on the map below, drag to look, tappable walk arrows to neighbours, a time-travel picker for historical captures, and a full-screen toggle. Attribution (address, capture date, "(c) Google") shown. **Gated OUT of the restricted flavor** (`!RESTRICTED_BUILD`): it still fetches from googleapis, so the content-minimal build omits the pill entirely. Device-verified end to end in Philadelphia (metadata, tile render, drag, walk, time-travel), restricted flavor confirmed to hide it. **Fully D-pad operable (fork-only):** the pano is a focusable that engages on OK - arrows then look around, +/- zoom, OK walks to the neighbour you face, BACK disengages; disengaged, arrows cycle a focus ring through Close/Fullscreen/Time-travel (which overlap the sphere, so an explicit requester ring, not spatial nav), and the date list is arrow-traversable. A real browser declares CATEGORY_APP_BROWSER or is any user-installed https handler; scheme-less website values (bare domains from Google's payload) count as web URLs. STRICTLY link buttons: the map, every in-app WebView (photos, reviews, transit, departures), voice downloading, the updater and tel:/geo: are untouched and fully functional on a browser-less phone. - [x] **Every exit card shows the direction of its exit (issue #79).** The reporter saw one identical symbol on every exit card. The cause was in the data, not the drawing: on live routes Google's keyless feed sends `maneuver='OFF_RAMP'` with **no** child `<turn side=>`, and the localized instruction carries no direction word either ("Take the exit toward &lt;road&gt;"), so the parser correctly fell through to the generic merge arrow - for left and right exits alike. Nothing in the response can fix that, because the direction only exists in the shape of the road. So Vela measures it: at each direction-less ramp/exit it compares the heading into the maneuver with the heading out of it, sampled ~60 m either side along the route geometry to skip the jitter right at the point, and reads the sign of the difference. Ramps are gentle (a motorway exit is often 15-25 degrees where a turn is 90), so the threshold is 12 degrees; below it the exit really does go straight ahead and keeps the merge arrow, which is the honest symbol for it. Only maneuvers the feed left undirected are touched - a real MERGE token, and any exit the feed DID label, are untouched. Device-verified on a Tel Aviv → Haifa route: the exits now draw the mirror of a feed-labelled RAMP_LEFT, i.e. right-hand exits, matching the road.

Position disambiguates: directly after the preposition that introduces the road name (`onto`/`on`/`toward`/`at`/`along`) the token is the title, so it reads Saint/Doctor; anywhere else it stays the suffix. Anchoring to the preposition rather than "followed by a capital letter" is what keeps "Bay St Ste 200" reading as Street. `Mt`/`Ft`/`Jr`/`Sr` are never road types, so they expand unconditionally (Mount/Fort/Junior/Senior). English only, like the rest of the table.

## Location (degoogled)

- [x] **Google-style location indicator** - browse: a blue dot with a heading cone driven by the rotation-vector compass (facing shown even standing still), greying when the fix is stale.
  - **Nav: a solid blue arrow puck** that snaps onto the route line and faces down the road, engaging only within ~22 m and heading the road's way, so a wrong-way/off-road fix shows the real position.
  - The puck rides a forward-progress motion model with dead reckoning, eased progress and smoothed heading, its speed from a Kalman filter fusing GPS speed with measured acceleration.
  - The on-route match is forward-only and monotonic within a bounded look-ahead, so a route passing near itself can't pull the puck backward; it holds and dead-reckons when nothing is ahead, re-acquiring globally only at start or after a reroute.
  - The follow-camera tracks the same smoothed point and holds the road-aligned heading off-route; the traversed-grey split rides the puck's drawn position as a hard cut.
  - Heading and speed are synthesised from movement when a fix lacks them, gated on real movement; single-fix speed spikes and position outliers are rejected and position runs through a speed-adaptive low-pass, with re-anchoring on a persistent leap.
